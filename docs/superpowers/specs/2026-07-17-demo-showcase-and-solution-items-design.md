# Design: browser demo showcase + solution items

**Date:** 2026-07-17
**Status:** approved (design), pending implementation plan
**Context:** Phase 3 §6 gate is closed (DECISIONS #80) and the first RADICAL increment `@if`
is merged (DECISIONS #81). There is no committed, human-facing way to *see* the compiled apps
run — `bench/publish/` exists only for measurement (minified, gzip siblings, parity checks) and
is gitignored. This adds a browsable demo, and surfaces the top-level docs in the IDE.

## Goal

Two deliverables, one change set:

1. A `demo/` folder — a static, browser-runnable showcase of **Counter**, **Rows**, and **If**,
   each mounting the JS that **`Filament.Generator` emits from `.razor`** (not the hand-written
   answer keys). It reuses the repo's existing mount contract and bundling path; it invents no new
   runtime or mount mechanism.
2. `Filament.sln` gains two solution folders so the docs and demo are visible in the IDE:
   **Solution Items** (`README.md`, `DECISIONS.md`, `BENCH.md`, `.gitignore`) and **demo**
   (`demo/index.html`, `demo/build.sh`).

This is scaffolding + a viewer. It changes **no emitted byte** of the generator or runtime, and it
is **not a measurement** — the honesty guards below keep a demo bundle from ever being mistaken for
the C1 artifact.

## Decisions taken during brainstorming

1. **Demo kind = static browser page**, not a console runner and not the Blazor baselines. Seeing
   the compiled apps run in a browser is the point; a console dump of JS is not a demo.
2. **Apps = Counter + Rows + If.** Counter is the canonical simple app, Rows exercises the keyed
   `list()` path, and If is the newest increment (whose `@if` *reuses* `list()`). All three already
   exist as samples.
3. **Mount source = generator output.** The demo compiles the `.razor` sources with
   `Filament.Generator` and mounts the emitted JS — the honest "this is what the compiler produces"
   demo, on-brand with the `-gen` bench labels. Mounting the hand-written answer keys was rejected:
   it would demo hand-written JS, not the compiler.
4. **Openability = build step, gitignored output.** `demo/build.sh` produces a dev bundle into a
   gitignored `demo/dist/`. Committing prebuilt bundles was rejected: it breaks the repo's core
   rule that **nothing runnable is committed** — `dist/`, `bench/publish/`, and every `.g.js` are
   gitignored and rebuilt from source precisely because a committed build artifact is one somebody
   eventually hand-edits (DECISIONS #58 and the `.gitignore` prose spell this out).

## Architecture

### Reuse the existing mount contract — do not invent one

Every runnable app in this repo is already an ES-module `main.js` that imports its generated
`.g.js` and calls `mount(document.getElementById('app'))` (see `samples/filament-counter-gen/main.js`).
The demo reuses that contract exactly:

- **Counter** → reuse `samples/filament-counter-gen/main.js` (imports `./Counter.g.js`), compiled
  from `samples/Counter/Counter.razor`.
- **Rows** → reuse `samples/filament-rows-gen/main.js` (imports `./Rows.g.js`), compiled from
  `baseline/Rows.Blazor/RowsApp.razor` (the file Blazor itself compiles — no drifting copy).
- **If** → add one new, parallel, committed entry `samples/filament-if-gen/main.js` (imports
  `./If.g.js`), compiled from `samples/If/If.razor`. `samples/filament-if-gen/If.g.js` is added to
  `.gitignore` exactly like the other two `.g.js` files.

Reusing the existing `-gen` entries means there is **one** set of generator-output entry modules,
shared by bench and demo — no second copy to drift.

### `demo/build.sh` — dev-mode sibling of `build-filament.sh`

Structurally mirrors `bench/build-filament.sh` (same rm-then-emit-then-verify discipline) but in a
**dev mode** distinct from the measurement mode:

For each of the three apps:
1. **Generate:** `rm` the co-located `.g.js`, then `dotnet run --project src/Filament.Generator -c
   Release -- <razor-src> <g-js-out>`. Verify from the artifact (file exists, carries the
   `GENERATED by Filament.Generator` banner), never from the exit code — the same stance
   `build-filament.sh` takes. If the generator refuses (a construct outside the subset), the build
   fails loudly with the diagnostic; that is the generator working, not failing.
2. **Bundle (dev):** `esbuild <entry> --bundle --format=iife --target=es2022
   --define:__FILAMENT_STATS__=false --outfile=demo/dist/<app>/app.js`. **Unminified, no
   `--drop:console`, no gzip siblings** — this is a viewer, not a weighed artifact.
3. **Shell + style:** write a per-app `demo/dist/<app>/index.html` (mount point `<div id="app">`
   + `<script src="app.js">`), and copy the app's stylesheet (Counter → Counter.Blazor's
   `app.css`, Rows → Rows.Blazor's `app.css`, If → a minimal/empty sheet, since there is no
   baseline `If.Blazor`).

`demo/dist/` is gitignored. `esbuild` and `dotnet` are already repo dependencies.

### `demo/index.html` — the committed landing page

A single self-contained page:
- Short Filament framing (one paragraph: "these apps are running from JS the compiler emitted from
  `.razor`, on a sub-2 KB runtime").
- A tab bar: **Counter / Rows / If**.
- An `<iframe>` that loads the selected app's built page from `demo/dist/<app>/index.html`.
  Iframes give clean isolation, so Counter's and Rows' different stylesheets never collide, and each
  app page is exactly the shell `build.sh` produced.
- Next to each tab, the app's `app.js` byte size, **fetched client-side** (a `fetch()` on the
  bundle, `.blob().size`) so the committed `index.html` stays static and never encodes a per-build
  number. Reinforces the sub-10 KB thesis.

**Styling:** self-contained inline CSS — no CDN, no build. This is a deliberate, documented
deviation from the "use Tailwind" global guideline: the demo must run with zero external
dependencies, and that guideline targets Blazor component work, not a standalone static page.

### Data flow

```
.razor  --Filament.Generator-->  .g.js  --esbuild (dev)-->  demo/dist/<app>/app.js
                                                                     |
                                              loaded by demo/dist/<app>/index.html
                                                                     |
                                                        mount() into <div id="app">
                                                                     |
                                       demo/index.html composes the three via <iframe>
```

## Honesty guards (this is a demo, never a measurement)

- The dev bundle is **unminified and ships no gzip siblings**, so it is structurally not the C1
  bundle and cannot be confused for one.
- `build.sh` prints, on every run, that C1/C3/C4 weights come from `bench/build-filament.sh`
  and `bench/harness/bench.mjs` — never from `demo/`.
- `.g.js` files are `rm`-then-emitted every build (staleness impossible), matching the existing
  generator-output rule.

## Solution wiring (`Filament.sln`)

Hand-edit the classic-format `.sln` (it is not `.slnx`) to add two **solution folders** — project
type GUID `{2150E333-8FDC-42A3-9474-1A3956D46DE8}`, the standard VS solution-folder type — with
fixed IDs consistent with the existing `{6E1B0A11-…}` scheme:

- **Solution Items** — `ProjectSection(SolutionItems)` listing `README.md`, `DECISIONS.md`,
  `BENCH.md`, `.gitignore`.
- **demo** — `ProjectSection(SolutionItems)` listing `demo\index.html`, `demo\build.sh`.

Solution folders carry no build configuration, so **no** `ProjectConfigurationPlatforms` entries
are added; the existing generator/test project config blocks are untouched. `dotnet sln` does not
manage solution-folder files, so this is a careful hand-edit — verified afterward by
`dotnet sln Filament.sln list` (must still list the two real projects) and `dotnet build`.

## Testing / verification

The gate is that the **compiled output actually runs**, not merely that files were produced:

1. `demo/build.sh` runs green (generator emits all three `.g.js`; esbuild bundles all three).
2. Serve `demo/` (e.g. `python3 -m http.server`) and load each app in a headless browser
   (chrome-devtools MCP), asserting real behavior:
   - **Counter** — click `#increment`, assert `#counter-value` advances.
   - **Rows** — trigger create, assert rows render into the table body.
   - **If** — click `#t`, assert `#msg` (`<span>hi</span>`) toggles in/out.
3. `dotnet sln Filament.sln list` still lists `Filament.Generator` and `Filament.Generator.Tests`;
   `dotnet build` succeeds; the existing test suite stays green (this change adds no C# and touches
   no runtime source, so 172/0 must hold).

## Explicitly out of scope

- **The `@if` family increment** (`@else`/`@else if`, nesting, root-level `@if`, multi-node body,
  next-sibling anchoring). That is the chosen next *feature* increment and gets its own spec/plan;
  this demo shows only the current simple `@if`.
- **Adding the baseline Blazor projects** (`baseline/Counter.Blazor`, `baseline/Rows.Blazor`) or
  the empty `src/Filament.Core` / `src/Filament.Analyzer` dirs to the solution. Offered separately.
- **Any change to the generator, runtime, or bench** beyond adding the one new `samples/filament-if-gen/`
  entry module and its `.gitignore` line.
